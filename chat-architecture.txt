STATE INTERNO:
  messages: Message[]           = []
  conversationId: string|null   = loadConversationId()  (localStorage)
  pending: boolean              = false
  error: string|null            = null
  loadingHistory: boolean       = false
  contextToggles: ContextToggle = { rag_chunks:true, indexed_assets:true,
                                    graph_nodes:true }
  webSearchEnabled: boolean     = loadWebSearchEnabled() (localStorage)

PERSISTENCIA:
  localStorage("geonexus.activeConversationId") — ID de conversación activa
  localStorage("geonexus.webSearchEnabled")     — Estado de búsqueda web

COMPORTAMIENTO DE submit(content):
  1. Validar: !pending, content.trim(), activeProvider
  2. Optimistic user message:
     id="local-{Date.now()}", role="user", content=clean
  3. Construir SendMessageInput con contextToggles y webSearch
  4. Si webSearchEnabled:
     ─ Crear placeholder assistant:
       id="research-{Date.now()}", isSearching=true, researchSources=[]
     ─ Iniciar timer cada 500ms que actualiza:
       searchElapsedSeconds, currentSearchQuery
         <2s:  query original
         <4s:  query + "análisis de resultados"
         >=4s: query + "generando respuesta"
  5. await sendMessage(input)  →  invoke("send_message", { input })
  6. Éxito:
     ─ Si webSearch: updateAssistantMessage(placeholderId, {
         isSearching:false, researchSources:response.research_sources,
         content:response.message.content, ...})
     ─ Si no webSearch: agregar response.message a messages[]
  7. Error:
     ─ Filtrar mensajes optimistas
     ─ Mostrar toast error
  8. finally: setPending(false)

COMPORTAMIENTO DE regenerate():
  1. Buscar el último user message en messages[]
  2. setMessages: eliminar desde ese user message hasta el final
  3. submit(lastUserContent) — re-envía el mensaje

RETURN:
  { activeProvider, conversationId, error, messages, pending,
    loadingHistory, contextToggles, setContextToggles,
    webSearchEnabled, setWebSearchEnabled, submit, regenerate,
    loadConversation, newConversation }

5b. send_message.rs   (308 lines) — CORAZÓN DEL CHAT
────────────────────────────────────────────────────────────────────────────────
  #[tauri::command] send_message(state, input) → SendMessageResponse

  FLUJO COMPLETO:
    1. Validar input (input.validate())
    2. Generar trace_id (UUID v4)
    3. Asegurar conversación (ensure_conversation)
    4. Crear user Message, insertar en DB (chat_repo::insert_message)
       ─ Si nueva conversación: generar título (title_from_message)
    5. RAG context:
       ─ run_sidecar_json::<Vec<RecallChunk>>(["--action","recall_chunks",
           "--query",input.content,"--project_id",input.project_id,"--top_k","4"])
       ─ Formatear como contexto textual "[1] texto..."
    6. Project context:
       ─ run_sidecar_json::<Value>(["--action","build_project_context",
           "--project_id",input.project_id])
       ─ Extraer "context" string
    7. Web search (si input.web_search):
       ─ extract_search_query(input.content) — limpia query
       ─ run_sidecar_json::<Vec<ResearchSource>>(["--action","search_web",
           "--query",search_query,"--max_results","5"])
       ─ Formatear como contexto "[Fuente 1] title - url: snippet"
    8. Tool definitions:
       ─ get_tool_definitions() → 4 tools (read_file, search_code,
         list_directory, glob_files)
    9. Build messages array:
       ─ chat_repo::list_messages (historial)
       ─ context::build_messages(history, project_context, web_context,
         rag_context, user_content)
       ─ Sistema: "Eres GeoNexus IA..." + contextos + historial + msg actual
   10. Tool-calling loop (máx 10 iteraciones):
       ─ run_sidecar(["--action","chat_llm","--provider_type",...,
           "--messages",messages_json,"--tools",tools_json])
       ─ Parsear SidecarChatResult
       ─ Si tiene tool_calls:
           push assistant(tool_calls) + tool(result)
           execute_tool_call(tc)
           continue
       ─ Si no: extraer final_content, break
   11. Guardar assistant Message:
       ─ sources = recall_chunks[].source
       ─ research_sources = si hay resultados de búsqueda
       ─ chat_repo::insert_message
   12. Return SendMessageResponse {
         conversation_id, message, chunks_used: vec![], trace_id,
         research_sources, search_query
       }

  FUNCIONES AYUDANTES:
    ensure_conversation() — devuelve conversation_id existente o crea nueva
    extract_search_query() — limpia query:
      ─ Si <60 chars: devuelve tal cual
      ─ Si >=60: quita prefijos (dime, cuéntame, qué es, cómo, etc.)
      ─ Trunca a 80 chars
      ─ Capitaliza primera letra

create_conversation(db, project_id, workspace_id, provider, model)
  INSERT INTO conversations ...
  → Conversation

list_conversations(db, project_id)
  SELECT c.*, (SELECT COUNT(*) FROM messages m WHERE m.conversation_id=c.id)
    AS message_count FROM conversations c WHERE project_id=? ORDER BY updated_at DESC
  → Vec<Conversation>

insert_message(db, msg: &Message)
  INSERT INTO messages (id, conversation_id, role, content, provider, model,
    trace_id, chunks_used, nodes_used, tool_calls, sources_json,
    research_sources, created_at)
  -- chunks_used, nodes_used, tool_calls → JSON string
  -- sources_json → JSON string
  -- research_sources → JSON string | NULL
  -- UPDATE conversations SET updated_at = ? WHERE id = ?
  → Result<(), String>

list_messages(db, conversation_id)
  SELECT m.*, m.sources_json AS sources
  FROM messages m WHERE conversation_id=? ORDER BY created_at ASC
  → Vec<Message> (via row_to_message)

row_to_message(row):
  ─ role: string → wrap en quotes → serde_json → MessageRole
  ─ chunks_used, nodes_used, tool_calls, sources: JSON string → vec
  ─ research_sources: JSON string → Option<Vec<ResearchSource>>
    (None si vacío)

delete_conversation(db, id):
  DELETE FROM messages WHERE conversation_id=?
  DELETE FROM conversations WHERE id=?

update_conversation_title(db, id, title):
  UPDATE conversations SET title=?, updated_at=? WHERE id=?

update_conversation_timestamp(db, id):
  UPDATE conversations SET updated_at=? WHERE id=?

║                 13. NOTAS IMPORTANTES                                       ║
╚══════════════════════════════════════════════════════════════════════════════╝

SERDE: El Rust Message struct NO tiene #[serde(rename_all = "camelCase")].
  Los campos se serializan en snake_case. El frontend SendMessageResponse
  también usa snake_case (research_sources), pero el frontend Message type
  usa researchSources (camelCase). Esto funciona porque:
  ─ Para web search: el hook usa response.research_sources (top-level)
    y lo mapea explícitamente a researchSources en el placeholder.
  ─ Para mensajes históricos (listMessages): el Message de Rust se serializa
    como research_sources pero el frontend espera researchSources.
    Si se carga historial, message.researchSources será undefined.
    → Esto es un bug conocido pero no crítico (los mensajes históricos
      con web search no muestran el panel de fuentes).

PYTHON VENV: El Rust busca Python en este orden:
  1. ai/.venv/Scripts/python.exe
  2. .venv/Scripts/python.exe
  3. ai/.venv/bin/python
  4. .venv/bin/python
  5. "python" (sistema)

  Si necesitas que un paquete Python funcione en el chat, debe instalarse
  en ai/.venv/ (el más prioritario).

DEPENDENCIAS PYTHON DEL CHAT:
  ─ ddgs: búsqueda DuckDuckGo (ai/.venv/)
  ─ requests: fallback DuckDuckGo Lite, Google, Bing, SerpAPI
  ─ httpx, lxml, primp: requeridos por ddgs

TOOL-CALLING LOOP: El LLM puede llamar herramientas (read_file, search_code,
  list_directory, glob_files) hasta 10 veces en una sola interacción. Cada
  iteración ejecuta el sidecar Python con chat_llm.

ESTADOS POSIBLES DEL CHAT:
  ─ Normal: messages.length > 0, !pending
  ─ Cargando historial: loadingHistory=true
  ─ Enviando mensaje: pending=true (ThinkingInline visible)
  ─ Deep Research: pending=true + placeholder con isSearching=true
    (DeepResearchPanel visible con spinner)
  ─ Error: error !== null (toast + mensaje en el composer)
