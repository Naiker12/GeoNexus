GeoNexus — Funciones pendientes por implementar (análisis detallado)
======================================================================
Fecha: 2026-06-09

Este documento cataloga TODO lo que falta implementar en GeoNexus,
organizado por prioridad y módulo. Incluye bugs activos, stubs,
placeholders, código muerto y funciones declaradas pero no implementadas.

======================================================================
1. BLOQUE E — Tool-calling loop en send_message (PRIORIDAD #1)
======================================================================

Actualmente send_message (chat.rs) envía el mensaje al LLM y devuelve la
respuesta, pero NO ejecuta un loop de tool-calling. El LLM puede pedir
ejecutar herramientas pero el sistema no las despacha.

Funciones faltantes:
- dispatch_tool_call(tool_name, args) integrado en send_message
- Reconocer tool_calls en la respuesta del LLM y ejecutar las tools
  containers-mcp (container_list, container_search, container_get)
- Loop de hasta N iteraciones (tool_call → ejecutar → resultado al LLM
  → siguiente turno) con límite configurable (hoy hardcoded max_iter=10)
- trace_id por mensaje y por tool-call para trazabilidad
- Manejo de errores de tools (timeout, permiso denegado, tool no encontrada)
  sin romper la conversación
- Registrar tool_calls en la tabla chat_tool_calls (migración ya existe)

Impacto: Sin esto, el chat no puede ejecutar acciones reales como listar
archivos, buscar documentos o sincronizar conectores desde la conversación.

4. STREAMING DE RESPUESTAS LLM
======================================================================

Todo el chat es request/response bloqueante. No hay streaming.

Faltante:
- stream_chat_response(conversation_id, message_id) como comando Tauri
- Soporte SSE en sidecar Python para todos los proveedores

5. BUGS ACTIVOS — Queries rotas e inconsistencias
======================================================================

5.1 ChromaDB collection name mismatch — CRÍTICO
-------------------------------------------------
- indexer.py escribe embeddings en colección "project_{id}_assets"
- sidecar.py recall_chunks consulta colección "project_memory" (default)
- RESULTADO: Los chunks indexados NUNCA se encuentran en el recall.
  El RAG siempre devuelve 0 resultados.

5.2 context/builder.py query column name — CRÍTICO
--------------------------------------------------
- builder.py:35 consulta "SELECT label, kind FROM graph_nodes"
- La columna real en SQLite es "name", no "label"
- RESULTADO: builder.py SIEMPRE lanza excepción y devuelve string vacío.
  Además el except Exception: pass lo silencia completamente.

5.3 context/builder.py status filter — CRÍTICO
-------------------------------------------------
- builder.py:21 filtra "status = 'indexed'"
- El valor real en SQLite para AssetStatus::Ready es "ready"
- RESULTADO: El filtro nunca encuentra assets, el contexto del proyecto
  siempre está vacío.

5.4 Anthropic model lists divergentes
---------------------------------------
- Python anthropic.py (hardcoded): claude-opus-4-1, claude-opus-4,
  claude-sonnet-4, claude-haiku-3
- Rust llm.rs (hardcoded): claude-opus-4, claude-sonnet-4, claude-haiku-3,
  claude-3-haiku (diferentes nombres y cantidades)
- Necesitan sincronizarse desde una fuente única

5.5 MCP container_get inconsistencia
--------------------------------------
- Rust containers_mcp.rs pasa file_id como ID de base de datos (entero)
- Python handlers.py _handle_get trata file_id como ruta de archivo
  y llama guard.resolve_local_path(file_id)
- Los dos componentes no son compatibles

19. PYTHON SIDECAR — Problemas de robustez
======================================================================

- timeout embedding Ollama = 1 segundo (efectivamente siempre falla)
- timeout embedding OpenAI = 5 segundos (muy corto para textos grandes)
- OpenAI embedding sin batching (cada chunk = 1 request HTTP)
- router.py no pasa api_key a ping_llm ni list_llm_models
- Anthropic no soportado en chat_llm_provider (solo ping)
- router.py no usa los módulos dedicados (openai.py, anthropic.py, etc.)
- MockChromaClient no es thread-safe (JSON sin lock)
- MockChromaCollection no implementa delete()
