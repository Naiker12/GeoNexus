GeoNexus - Plan backend para Datos, Conectores y memoria
Fecha: 2026-06-07
Estado: guia de trabajo V2 beta

1. Objetivo del documento

Este documento resume como se debe integrar la pagina Datos con el backend de
GeoNexus, tomando como base la arquitectura V2 del PDF, la UI actual y las
buenas practicas que se quieren mantener en el proyecto.

La idea principal:

- Conectores: fuentes externas o locales de archivos.
- Documentos: entrada documental y extraccion de texto.
- Datos: centro operativo donde se ve inventario, cache, embeddings, sync,
  ChromaDB y Knowledge Graph.
- MCP: herramientas que la IA puede ejecutar, por ejemplo QGIS, memoria,
  contenedores y Supabase/Postgres.
- Grafo: relaciones entre normas, documentos, capas, zonas y conceptos.

La pagina Datos no debe reemplazar Documentos ni Conectores. Debe mostrar que
paso con cada archivo o asset despues de entrar al sistema.


2. Funciones actuales de frontend

2.1 Chat IA

Responsabilidad:
- Conversacion con GeoNexus IA.
- Seleccion futura de modelos/proveedores.
- Punto final donde se usan documentos, capas, MCP tools y memoria.

Backend futuro:
- send_message(project_id, content, llm_config)
- stream_chat_response(trace_id)
- dispatch_tool_call(tool_name, args)
- store_conversation(project_id, messages)

2.2 Documentos

Responsabilidad:
- Subir documentos.
- Conectar fuentes documentales.
- Mostrar documentos listos, indexando o pendientes.
- Preparar extraccion de texto, chunks y citas.

Backend futuro:
- add_document(project_id, source, file_ref)
- extract_document_text(document_id)
- create_document_chunks(document_id)
- index_document(document_id)
- get_document_status(document_id)

2.3 Conectores

Responsabilidad:
- Mostrar proveedores disponibles: OneDrive, SharePoint, Google Drive,
  Dropbox, carpeta local, QGIS, ArcGIS Pro, APIs externas, S3/MinIO.
- Simular flujo de permisos, herramientas MCP e indexacion.
- No conectar bases de datos directas aqui.

Nota:
- PostGIS, PostgreSQL y Supabase deben vivir como MCP/data backend, no como
  conector visual de archivos.

Backend futuro:
- list_connector_providers()
- start_connector_auth(provider)
- complete_connector_auth(provider, callback_url)
- register_connector(project_id, provider, config)
- list_connector_files(connector_id, path)
- sync_connector(connector_id, remote_path, local_path)

2.4 Datos

Responsabilidad:
- Mostrar inventario de assets indexados.
- Mostrar cache local, estado de sync, chunks, embeddings y nodos del grafo.
- Mostrar lineage: conector -> cache -> indexador -> ChromaDB ->
  Knowledge Graph -> GeoNexus IA.
- Auditar si un archivo esta listo para IA o falta OAuth/sync/indexacion.

Backend futuro:
- list_data_assets(project_id)
- get_data_asset(asset_id)
- get_data_lineage(asset_id)
- get_data_store_metrics(project_id)
- get_sync_events(project_id)
- reindex_asset(asset_id)
- validate_data_asset(asset_id)

2.5 Servidores MCP

Responsabilidad:
- Mostrar MCP servers y tools.
- Simular QGIS MCP, Memory MCP, ArcGIS MCP y Supabase MCP.
- Supabase/Postgres se usa aqui para consultar o guardar informacion mediante
  tools controladas.

Backend futuro:
- list_mcp_servers()
- register_mcp_server(config)
- ping_mcp_server(server_id)
- list_mcp_tools(server_id)
- dispatch_mcp_tool(server_id, tool_name, args)

2.6 Grafo

Responsabilidad:
- Visualizar relaciones entre documentos, normas, capas GIS, zonas y conceptos.
- Mostrar detalle de nodos, evidencia y relaciones.

Backend futuro:
- list_graph_nodes(project_id)
- list_graph_edges(project_id)
- get_graph_node(node_id)
- rebuild_knowledge_graph(project_id)

4. Buenas practicas obligatorias

4.1 Frontend

- Un componente por archivo cuando el bloque tenga responsabilidad propia.
- Evitar archivos gigantes. Objetivo: idealmente menos de 300 lineas.
- Data mock separada de componentes visuales.
- Estado minimo en el padre. Derivar datos cuando sea posible.
- No poner logica de negocio real en React.
- UI solo renderiza estado y dispara acciones.
- Dialogos y sheets deben tener scroll interno si el contenido crece.
- Botones con iconos claros y texto corto.
- No duplicar pantallas: Datos audita, Documentos ingresa, Conectores conecta.

4.2 Backend Rust/Tauri

- Tauri commands validan inputs antes de llamar al core o sidecar.
- Nada de panic en produccion.
- Usar Result y errores tipados.
- No exponer rutas fuera de allowlist.
- No loggear tokens, API keys ni callbacks completos.
- Cada command debe incluir project_id cuando afecte datos del proyecto.
- Mantener dependencias unidireccionales:
  React -> Tauri command -> core/db/mcp/sidecar.

4.3 Python sidecar

- Type hints en funciones publicas.
- Providers con interfaz comun.
- Errores tipados: NotFound, AuthExpired, QuotaExceeded, NetworkError.
- No guardar secretos en SQLite.
- Cachear archivos descargados con metadata de ETag o modifiedTime.
- Separar router, provider, cache y sync.

4.4 Seguridad

- OAuth con PKCE para proveedores cloud.
- Tokens en keychain o Stronghold.
- SQLite solo guarda metadata sin secretos.
- Scopes minimos por proveedor.
- Subidas automaticas desactivadas por defecto.
- Confirmacion del usuario antes de modificar o subir archivos.
- Logs con trace_id, provider_id y project_id, nunca tokens.


5. Primer backend que conviene construir

Antes de OAuth real, conviene construir una base funcional y segura:

Fase 1: inventario y metadata local

Objetivo:
- Que la pagina Datos ya pueda leer datos reales desde backend, aunque los
  conectores cloud sigan simulados.

Funciones iniciales:

1. list_data_assets(project_id)
   Devuelve assets con:
   - id
   - name
   - kind
   - source
   - location
   - status
   - updated
   - size
   - chunks
   - embeddings
   - graph_nodes
   - cache_state

2. get_data_asset(asset_id)
   Devuelve detalle de un asset y su lineage.

3. get_data_store_metrics(project_id)
   Devuelve estado de:
   - SQLite metadata
   - Cache local
   - ChromaDB
   - Knowledge Graph

4. get_sync_events(project_id)
   Devuelve eventos simulados o reales del pipeline.

5. validate_data_asset(asset_id)
   Revisa:
   - existe archivo o referencia
   - ruta permitida
   - metadata coherente
   - cache vigente
   - chunks generados

Por que esta fase primero:
- Permite conectar la pagina Datos sin esperar OAuth.
- Da estructura real a SQLite.
- Permite probar UI, estados, errores y trazabilidad.
- Prepara Documentos, Conectores, Grafo y Chat para compartir la misma verdad.


6. Segundo bloque backend

Fase 2: conector local y cache

Objetivo:
- Implementar carpeta local/red como primer conector real.
- Evitar OAuth al inicio.
- Probar allowlist, lectura de archivos y cache.

Funciones:
- register_local_connector(project_id, root_path)
- list_connector_files(connector_id, path)
- get_connector_file(connector_id, file_id)
- cache_connector_file(connector_id, file_id)
- sync_local_connector(connector_id)

Tablas SQLite sugeridas:
- projects
- connector_configs
- connector_files
- data_assets
- sync_events
- document_chunks
- graph_nodes
- graph_edges


7. Tercer bloque backend

Fase 3: indexacion documental y vectorial

Objetivo:
- Convertir documentos reales en chunks y embeddings.

Funciones:
- extract_document_text(document_id)
- create_document_chunks(document_id)
- embed_chunks(document_id)
- upsert_chroma_vectors(document_id)
- link_chunks_to_asset(asset_id, chunk_ids)

Resultado:
- Datos muestra chunks y vectores reales.
- Chat puede hacer recall.
- Grafo puede crear relaciones basicas.


8. Cuarto bloque backend

Fase 4: containers-mcp

Objetivo:
- Exponer conectores como tools para la IA.

Tools:
- container_list(provider, path)
- container_get(provider, file_id)
- container_search(provider, query)
- container_sync(provider, remote, local)
- container_upload(provider, path, local_path)

Reglas:
- container_upload siempre pide confirmacion.
- container_get valida allowlist y permisos.
- container_search no debe leer contenido sensible sin permiso.


9. Quinto bloque backend

Fase 5: OAuth OneDrive

Objetivo:
- Primer proveedor cloud real.

Flujo:
1. React llama start_connector_auth("onedrive")
2. Tauri solicita auth_url al sidecar
3. Tauri abre browser del sistema
4. Microsoft retorna deep link geonexus://oauth/callback
5. Tauri captura callback
6. Sidecar valida state y cambia code por token
7. Token se guarda en keychain/Stronghold
8. SQLite guarda metadata del conector sin secretos
9. UI actualiza estado conectado

Scopes recomendados:
- Files.ReadWrite.Selected
- offline_access


10. Supabase/Postgres

Supabase/Postgres no debe estar como conector normal de archivos.
Debe vivir en MCP/Datos.

Uso:
- Consultar schemas
- Ejecutar SQL controlado o read-only
- Guardar contexto del proyecto si el usuario lo configura
- Auditar seguridad, RLS y permisos

Tools sugeridas:
- supabase_list_tables(schema, project_ref)
- supabase_execute_sql(sql_readonly, params, project_ref)
- supabase_store_context(workspace_id, content, metadata)
- supabase_security_advisors(schema, project_ref)

Buenas practicas:
- read_only=true por defecto.
- project_ref allowlist.
- No ejecutar SQL libre desde el chat sin validacion.
- Mostrar confirmacion antes de escribir datos.

13. Resultado esperado

Al terminar la primera etapa, GeoNexus debe poder:

- Mostrar assets reales en Datos.
- Saber de donde viene cada archivo.
- Saber si esta indexado, pendiente o con conflicto.
- Mostrar cache y eventos de sync.
- Preparar el camino para que Chat IA use archivos y capas con trazabilidad.

El objetivo no es conectar todo al mismo tiempo. El objetivo es crear primero
una columna vertebral limpia: metadata, inventario, cache, indexacion y lineage.
