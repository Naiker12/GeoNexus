GeoNexus - Estado de funciones implementadas vs plan backend Datos
Fecha de revision: 2026-06-08
Archivo base analizado: docs/plan_backend_datos_geonexus.txt

Resumen corto

El proyecto ya tiene implementada una base backend real para las fases 1, 2, 3
y 4 del plan. La fase 6 aparece iniciada en el codigo como integracion LLM,
pero todavia esta en modo chat simple: no usa streaming, memoria vectorial,
tool calls MCP, documentos, grafo ni conversaciones persistidas.

Estado general:
- Fase 1, inventario y metadata local: implementada en beta.
- Fase 2, conector local y cache: implementada en beta.
- Fase 3, indexacion documental y vectorial: implementada en beta.
- Fase 4, containers-mcp: implementada para proveedor local.
- Fase 5, OAuth OneDrive: pendiente.
- Fase 6, proveedor LLM base: iniciada y en correccion.
- Fase 7, chat real con memoria: Bloque A iniciado.

4. Diagnostico de avance real

Lo que ya tienes:
- Una base SQLite seria para inventario, conectores, archivos, eventos, chunks,
  grafo y auditoria MCP.
- Backend Tauri registrado en main.rs con comandos por fase.
- Conector local funcional en backend.
- Pipeline documental funcional en beta.
- ChromaDB o mock Chroma para persistir vectores.
- Grafo conectado a UI.
- Datos conectado a UI.
- Documentos conectado parcialmente a indexacion.
- containers-mcp funcional para proveedor local.
- LLM router basico con ping y chat simple.

Lo que todavia falta bastante:
- Cerrar la fase 6 de verdad: chat conectado, memoria, RAG, streaming, tools y
  persistencia.
- OAuth OneDrive/Fase 5.
- Conectar la pantalla Conectores al backend local.
- Convertir lineage de Datos en datos reales.
- Hacer que MCP sea una capa general y no solo containers-mcp local.
- Usar Supabase/Postgres como MCP/Datos read-only si se decide avanzar por esa
  parte.


5. Orden recomendado para continuar desde aqui

Prioridad 1: cerrar fase 6 minima
- Conectar ChatPanel con sendLlmMessage.
- Guardar mensajes en estado React.
- Seleccionar proveedor/modelo configurado.
- Mostrar loading/error/respuesta.

Prioridad 2: fase 6 con contexto real
- Crear endpoint/command de recall: buscar chunks en ChromaDB por project_id.
- Armar prompt con assets, chunks relevantes y nodos de grafo.
- Devolver respuesta con citas por asset/chunk.

Prioridad 3: fase 6 con tools
- Implementar dispatch_tool_call desde chat.
- Permitir container_list, container_search y container_get desde el chat.
- Registrar trace_id por mensaje y tool-call.

Prioridad 4: persistencia
- Crear tabla conversations/messages o equivalente.
- Implementar store_conversation(project_id, messages).
- Cargar historial por proyecto.

Prioridad 5: terminar conectores locales en UI
- Conectar ConnectorsPage/AddConnectorDialog con registerLocalConnector.
- Mostrar archivos del conector.
- Agregar botones sync/cache.

Prioridad 6: OAuth OneDrive
- Implementar Fase 5 despues de tener conector local y chat contextual estables.


Conclusion

GeoNexus ya tiene una columna vertebral backend bastante avanzada para Datos:
inventario, SQLite, conector local, eventos, indexacion, chunks, embeddings,
grafo y containers-mcp local. El punto mas incompleto frente al plan actual es
la fase 6: existe el inicio de LLM, pero falta convertirlo en chat real con
memoria, documentos, grafo, MCP tools, streaming y persistencia.
