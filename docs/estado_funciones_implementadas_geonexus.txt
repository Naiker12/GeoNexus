GeoNexus — Estado de implementación por fases (Junio 2026)
===========================================================

FASE 1 — Inventario y metadata local
Estado: ✅ Implementada en beta
Backend: list_data_assets, get_data_asset, get_data_store_metrics, get_sync_events, validate_data_asset operan sobre SQLite con tablas assets, sync_events y métricas agregadas. Frontend DataPage ya consulta estos comandos.
Pendiente: get_data_lineage dedicado, reindex_asset como comando separado, y LineagePanel dinámico.

FASE 2 — Conector local y cache
Estado: ✅ Implementada en beta
Backend: register_local_connector, list_connector_files, cache_connector_file, sync_local_connector funcionales con allowlist, diff, y registro en connector_configs/connector_files. 10 tests unitarios.
Pendiente: get_connector_file dedicado, copia física a cache cifrada, y conexión de la UI de Conectores al backend real.

FASE 3 — Indexación documental y vectorial
Estado: ✅ Implementada en beta
Pipeline completo: index_document orquesta sidecar Python → extract_text → chunk_text → generate_embeddings → upsert_chunks_to_vector_store → extract_graph_entities. ChromaDB con fallback MockChromaClient. Embeddings deterministas como fallback.
Pendiente: ChromaDB real garantizado en producción, citas por chunk/página en respuestas IA.

FASE 4 — Containers-MCP
Estado: ✅ Implementada para proveedor local
init_containers_mcp + dispatch_container_tool con 5 tools (list, get, search, sync, upload). container_upload reservado para Fase 5. Auditoría en container_mcp_calls. 8 tests unitarios.
Pendiente: Proveedores cloud (OneDrive, Google Drive, SharePoint, Dropbox, S3), integración con chat tool-calling (Bloque E).

FASE 5 — OAuth OneDrive
Estado: ❌ Pendiente
Nada implementado. Flujo requerido: start_connector_auth → abrir browser → capturar deep link geonexus://oauth/callback → validar state → cambiar code por token → guardar en keychain/Stronghold → registrar metadata sin secretos → sincronizar OneDrive real.

FASE 6 — Proveedor LLM base
Estado: ✅ Completada en beta
ping_llm_provider, send_llm_message, sidecar ping_llm/chat_llm. Router Python soporta Ollama, LM Studio, OpenRouter, OpenAI, Anthropic, custom. AiContainersPage permite probar conexión y configurar proveedores.
Pendiente: streaming de respuestas, dispatch de herramientas MCP desde chat.

FASE 7 — Chat real con memoria (Bloques A+B)
Estado: ✅ Completada
Tipos Conversation/Message en geonexus-core. Migración SQLite conversations + messages. chat_repo CRUD completo. send_message, list_conversations, list_messages. Frontend con ConversationList, ChatTranscript, historial persistido. 8 tests chat_repo + 14 tests chat commands.

FASE 8 — Auto-detect de modelos LLM
Estado: ✅ Completada
list_llm_models nativo en Rust para Ollama, LM Studio, OpenRouter, OpenAI, Anthropic, custom. ProviderSetupDialog con auto-detect, badges (free/context-length), debounce 600ms. 11 tests unitarios.

FASE 9 — RAG recall + fuentes + contexto GIS (Bloques C+D)
Estado: ✅ Completada
Migration chat_sources_tool_log.sql (sources_json + chat_tool_calls). recall_chunks vía sidecar con cosine similarity. send_message inyecta RAG recall + project context en system prompt. ProjectContextPanel con toggles de contexto. Sources en mensajes y badges. activeProvider retorna null si no hay modelo.
