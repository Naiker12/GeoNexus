  9. TYPES (src/types/analysis.ts)
═══════════════════════════════════════════════════════════════════════════════════

  Timeframe:    "hoy" | "7d" | "30d"  (union type)

  AnalysisMetrics:
    tokens_hoy: number       → Total de tokens consumidos hoje
    consultas_ia: number     → Quantidade de consultas ao assistente hoje
    costo_estimado: number   → Custo monetário estimado (USD)
    trazas_guardadas: number → Tool-calls registradas hoje

  TokenBucket:
    hora: string             → Rótulo da hora/dia (ex: "10:00", "12 jun")
    input_tokens: number     → Tokens de entrada
    output_tokens: number    → Tokens de saída
    total_tokens: number     → Soma input + output

  ModelUsage:
    model: string            → Nome do modelo (ex: "gpt-4o")
    provider: string         → Provedor (ex: "openai")
    tokens: number           → Total de tokens
    requests: number         → Quantidade de requisições
    input_tokens: number     → Tokens de entrada
    output_tokens: number    → Tokens de saída

  AnalysisRun:
    id: string               → ID único da execução
    conversation_id: string  → ID da conversação
    title: string            → Título/descrição
    model: string            → Modelo usado
    tokens: number           → Total de tokens
    duration_ms: number      → Duração em milissegundos
    trace_id: string | null  → ID de rastreamento (opcional)
    tool_calls: number       → Quantidade de tool calls
    created_at: number       → Timestamp (epoch seconds)

  SkillUsage:
    tool_name: string        → Nome da ferramenta (ex: "read_file")
    calls: number            → Quantidade de chamadas
    success_rate: number     → Taxa de sucesso (0-100)

  CostSummary:
    cost_hoy: number         → Custo do dia atual (USD)
    cost_7d: number          → Custo dos últimos 7 dias (USD)
    cost_total: number       → Custo total (USD)
    avg_per_query: number    → Custo médio por consulta (USD)

  TopQuery:
    title: string            → Texto da consulta
    runs: number             → Quantidade de execuções
    tokens: number           → Total de tokens usados

  11. DATA FLOW
═══════════════════════════════════════════════════════════════════════════════════

  Ciclo completo:

    1. AnalysisPage monta → cada hook dispara useEffect individualmente
    2. Cada hook chama sua API correspondente via invoke Tauri
    3. Backend Rust (analysis_repo.rs) consulta tabelas SQLite:
        - messages (input_tokens, output_tokens, total_tokens, model)
        - conversations (title, created_at)
        - chat_tool_calls (tool_name, success)
    4. Dados retornam → cada componente renderiza com data/loading/error
    5. Usuário interage:
        - Muda timeframe → TokenChart recarrega
        - Clica em modelo → ModelDetailDialog
        - Clica em traza → TraceDetailDialog
        - Exporta → CSV/JSON via API ou PNG/PDF via html-to-image

  Origem dos dados:
    ✅ messages.input_tokens, output_tokens, total_tokens (já existem desde send_message)
    ✅ conversations (para title, created_at)
    ✅ chat_tool_calls (para tool_name, success_rate)
    ❌ NENHUMA migração nova necessária

  Dados em tempo real:
    ❌ NÃO há eventos Tauri para análise (dados são consultados via invoke)
    ❌ NÃO há subscribe/websocket
    ❌ Refresh manual não existe (recarregar página recria hooks)

  13. EVENTOS (como funcionam)
═══════════════════════════════════════════════════════════════════════════════════

  A página de Análise NÃO usa eventos em tempo real.
  Diferente da DataPage (que tem SyncEvents via backend) ou do GraphPage
  (que escuta eventos Tauri graph:updated), a AnalysisPage funciona
  exclusivamente por consulta (polling no mount):

  Fluxo de eventos no ecossistema:
    ┌──────────────────────────────────────────────────┐
    │  send_message (chat)                             │
    │    → cria messages (input_tokens, output_tokens)  │
    │    → cria chat_tool_calls (tool_name, success)    │
    │    → atualiza conversations (title, timestamps)   │
    └──────────┬───────────────────────────────────────┘
               │
               ▼
    ┌──────────────────────────────────────────────────┐
    │  AnalysisPage monta                              │
    │    → 7 hooks disparam 7 consultas SQL            │
    │    → Dados já estão em SQLite desde send_message │
    │    → Sem eventos adicionais necessários          │
    └──────────────────────────────────────────────────┘

  Dados consultados:
    get_analysis_metrics → COUNT/SUM de messages hoje
    get_token_timeline   → GROUP BY hora/dia em messages
    get_model_usage      → GROUP BY model em messages
    list_analysis_runs   → SELECT com JOIN conversations
    get_skill_usage      → GROUP BY tool_name em chat_tool_calls
    get_cost_by_timeframe → SUM(cost) em messages
    get_top_queries      → GROUP BY title em conversations

  IMPORTANTE:
    ✅ Os eventos (mensagens, tool calls) são criados pelo sistema de chat
    ✅ A análise apenas CONSULTA esses dados já existentes
    ✅ Não há pipeline de eventos específico para análise
    ✅ Os dados são "frioz" (cold data) — consultados sob demanda
