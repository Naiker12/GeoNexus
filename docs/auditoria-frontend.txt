1. VISIÓN GENERAL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Frontend de escritorio construido con React 18 + TypeScript 5.4,
empaquetado con Vite 5, ejecutándose dentro de Tauri v2 como
aplicación de escritorio nativa.

No hay servidor web ni backend HTTP frontend — toda la comunicación
con Rust es via Tauri IPC (invoke). No hay React Router, React Query,
Redux, GraphQL, ni tRPC.

Arquitectura: Single-page application con hash-based routing custom.
Estado global con Zustand 5 + React Context. UI con Radix UI
primitives + Tailwind CSS v4 + Framer Motion.

6. API LAYER (src/api/)

16 módulos, todos con el mismo patrón:

1. Detectar runtime Tauri (__TAURI_INTERNALS__)
2. Import dinámico de @tauri-apps/api/core → invoke()
3. Llamar comando Rust por nombre string

Sin fetch/axios/GraphQL — 100% Tauri IPC.

9. ROUTING

Hash-based routing custom (sin react-router):

window.location.hash → activeRoute
  └─ #chat, #tasks, #memory, #files, #mcp, #uso,
     #conectores, #projects, #skills

Implementado en App.tsx via useHashRoute() hook:
- Lee window.location.hash (default "#chat")
- Escucha hashchange events
- GeoAgentsWorkspace.renderPage() hace switch

Ventaja: Simplicidad, sin dependencias extra.
Desventaja: Sin lazy loading, sin nested routes, sin guards.

El hook useReasoningTimeline centraliza la escucha de estos eventos
y expone timeline, isStreaming, thinkingText al resto de la UI.

16. DEBILIDADES / RIESGOS

✗ Sin lazy loading / code splitting (React.lazy / Suspense)
  → Todo el bundle se carga upfront (~2MB+ estimado)
✗ Sin React Router — hash routing custom limita nested routes,
  guards, y SSR/SSG
✗ Baja cobertura de tests (8 archivos para ~80 componentes)
✗ Sin tests de integración ni e2e (Playwright/Cypress)
✗ Sin React Query — cache manual, sin stale-while-revalidate,
  sin refetch automático
✗ Sin manejo de errores global (no hay error boundary por feature,
  solo un ErrorBoundary genérico en App)
✗ Sin loading skeletons para la mayoría de páginas
✗ Sin i18n (todo en español hardcodeado)
✗ Sin a11y audit (aria labels presentes pero no sistemáticos)
✗ Dependencia de Tauri API no abstraída — migrar a web requeriría
  reescribir toda la capa api/
✗ Sin WebSocket nativo en frontend (usa eventos Tauri como puente)
✗ 16 api modules con mucho boilerplate repetitivo
✗ Sin virtual scrolling (las listas de mensajes, nodos, etc.
  renderizan todo)
✗ Sonner para toasts es la única fuente de feedback al usuario

19. RECOMENDACIONES

Prioridad alta:
1. Agregar React.lazy + Suspense para code splitting por ruta
   (ChatPanel ~500KB, GraphPage ~300KB, etc.)
2. Agregar React Query o SWR para cache de API calls
   (evita llamadas repetitivas a invoke())
3. Aumentar cobertura de tests (mínimo: cada api module + cada hook)

Prioridad media:
4. Agregar error boundaries por feature (no solo uno global)
5. Agregar loading skeletons para páginas principales
6. Abstraer capa Tauri invoke para poder migrar a web si es necesario
7. Agregar virtual scrolling para listas largas (mensajes, nodos)

Prioridad baja:
8. Agregar i18n (inglés + español)
9. Auditoría a11y (aria labels, roles, focus management)
10. Reducir boilerplate en api/ con helper genérico
