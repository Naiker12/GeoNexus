1. VISTA GENERAL DEL SISTEMA
-----------------------------------------------------------------------------

El sistema de razonamiento es una arquitectura híbrida frontend-backend que permite:
- Visualización en tiempo real del proceso de razonamiento del agente
- Almacenamiento persistente de todos los eventos de razonamiento
- Jerarquía de eventos para mostrar relaciones padre-hijo
- Colapsado automático al finalizar la ejecución
- Tema oscuro estilo terminal profesional

2.1 src/hooks/useReasoningTimeline.ts
-----------------------------------------------------------------------------

ESTADO GESTIONADO POR EL HOOK:
  - timeline: ReasoningTimeline | null (para legacy)
  - traceEvents: AgentTraceEvent[] (nuevo sistema de eventos)
  - isStreaming: boolean (si está ocurriendo un razonamiento en tiempo real)
  - thinkingText: string (texto "pensando" en tiempo real)
  - isCollapsing: boolean (animación de colapsado)

FUNCIONAMIENTO:
1. Al montar el componente o cambiar sessionId, se inicializa el estado
2. Se suscribe a eventos Tauri:
   a. "reasoning:start":
      - Establece isStreaming=true
      - Restablece traceEvents
      - Establece activeSessionRef
   b. "agent:event":
      - Añade el evento a traceEvents
      - Actualiza la UI en tiempo real
   c. "reasoning:end":
      - Establece isStreaming=false
      - Establece duración total
      - Inicia temporizador para colapsar automáticamente (3 segundos)
3. Al cambiar sessionId, resetea TODO el estado
4. Cleanup: cancela listeners al desmontar

RETORNA:
{
  timeline,
  traceEvents,
  isStreaming,
  thinkingText,
  isCollapsing,
  toggleCollapse,
  collapseTimeline
}

DEFINICIÓN DE AgentTraceEvent (TypeScript):
interface AgentTraceEvent {
  type: string;                  // Tipo de evento: "started", "progress", "completed", "failed"
  id: string;                    // ID único del paso de razonamiento
  parent_id: string | null;      // ID del padre (para jerarquía)
  category: string;              // Categoría: "security", "agent", "mcp", "llm", "database", etc.
  title: string;                 // Título visible para el usuario
  log: string | null;            // Log interno del paso
  payload: any | null;           // Datos adicionales (JSON)
  duration: number | null;       // Duración en milisegundos
  user_friendly_summary: string | null; // Resumen para mostrar al usuario
  error: string | null;          // Error si ocurrió
  timestamp: string;             // Fecha en formato ISO
}

3.2 send_message.rs - crates/geonexus-tauri/src/commands/chat/send_message.rs
-----------------------------------------------------------------------------

ESTRUCTURA DEL COMANDO send_message():
1. Inicialización
   - Genera trace_id único (UUID)
   - Crea conversation_id
   - Inicializa trace_events: Vec<AgentTraceEvent> = vec![]
   - Emite "reasoning:start" a Tauri

2. Paso 1: Validación de entrada (Input Validation)
   - Emite AgentTraceEvent { type: "started", id: "validate", ... }
   - Realiza validación de la entrada
   - Emite AgentTraceEvent { type: "completed", id: "validate", ... }

3. Paso 2: Clasificación de intención (Intent Classification)
   - Emite started
   - Ejecuta classify_intent()
   - Emite completed con resultado
   - También emite legacy reasoning:step

4. Paso 3: Contexto de grafo (Graph Context)
   - started -> (consulta DB) -> completed
   - Carga datos relevantes del grafo

5. Paso 4: Búsqueda en vector DB (RAG Recall)
   - started -> ejecuta sidecar -> completed
   - Escanea para prompt injection!

6. Construcción de contexto y llamada a LLM
   - Crea history + context + rag
   - Llamada a sidecar para streaming

7. Loop de tool-calling (max 10 iteraciones)
   - Si LLM pide herramientas:
     - Ejecuta herramienta local o MCP
     - Emite eventos para cada paso
     - Añade respuesta al contexto

8. Finalización y guardado en DB
   - Emite reasoning:end
   - Convierte trace_events a JSON String
   - Guarda todo en SQLite en el campo messages.reasoning_events

FUNCIÓN AUXILIAR: emit_trace_event()
  Parámetros:
    - handle: Option<&Tauri::AppHandle>
    - event: AgentTraceEvent
    - events: &mut Vec<AgentTraceEvent>
  Funcionamiento:
    - Añade evento al vector local
    - Lo envía al frontend vía app.emit("agent:event", event)

4. FLUJO COMPLETO DE EVENTOS
-----------------------------------------------------------------------------

PASO A PASO DE UN RAZONAMIENTO NORMAL:

1. USUARIO ENVÍA UN MENSAJE
   → Frontend llama send_message() via Tauri command

2. BACKEND INICIA RAZONAMIENTO
   ├─ send_message() crea conversation_id y trace_id
   ├─ emit_reasoning_start(conversation_id)
   ├─ Frontend recibe "reasoning:start"
   └─ useReasoningTimeline establece isStreaming=true y borra traceEvents

3. EJECUCIÓN DEL FLUJO DE RAZONAMIENTO
   Por cada paso del razonamiento:
   a. Backend crea AgentTraceEvent { type: "started", ... }
   b. emit_trace_event() añade a Vec y envía a frontend
   c. useReasoningTimeline recibe "agent:event" y actualiza traceEvents
   d. TraceTreeView se actualiza en tiempo real
   e. Continúa con progreso y completado

4. EJEMPLO DEL FLUJO DEL BACKEND:

   emit_trace_event(handle, AgentTraceEvent {
       r#type: "started".into(),
       id: "validate".into(),
       parent_id: None,
       category: "security".into(),
       title: "Input Validation".into(),
       ...
   }, trace_events);
   ... realiza la validación ...
   emit_trace_event(handle, AgentTraceEvent {
       r#type: "completed".into(),
       id: "validate".into(),
       ...
   }, trace_events);

   - Luego "intent"
   - Luego "graph_context"
   - Luego "rag_recall"
   - Luego "generating" (llamada a LLM)
   - Y más pasos si hay tool-calls

5. RAZONAMIENTO TERMINA
   a. Backend emite "reasoning:end"
   b. Guarda mensaje con reasoning_events: Vec<AgentTraceEvent> en BD
   c. useReasoningTimeline recibe "reasoning:end" y isStreaming=false
   d. TraceTreeView inicia temporizador para colapsar en 3s
   e. Después de 3 segundos, isExpanded=false y se muestra solo el resumen

6. VISTA PERSISTENTE (cuando el usuario navega a un chat viejo)
   a. Frontend carga mensajes desde BD
   b. AssistantMessage recibe message.reasoning_events (Vec<AgentTraceEvent>)
   c. TraceTreeView muestra el árbol completo desde los datos almacenados

5. ESTRUCTURA DE DATOS: AgentTraceEvent
-----------------------------------------------------------------------------

CAMPOS DETALLADOS:

┌─────────────────────┬─────────────────────────────────────────────────────┐
│ CAMPO               │ DESCRIPCIÓN                                        │
├─────────────────────┼─────────────────────────────────────────────────────┤
│ type                │ Tipo de evento:                                    │
│                     │  - "started"      → inicio de un paso               │
│                     │  - "progress"     → progreso intermedio            │
│                     │  - "completed"    → paso finalizado correctamente │
│                     │  - "failed"       → paso falló                     │
├─────────────────────┼─────────────────────────────────────────────────────┤
│ id                  │ ID único del paso (para deduplicar)                 │
├─────────────────────┼─────────────────────────────────────────────────────┤
│ parent_id           │ ID del nodo padre (para jerarquía)                  │
│                     │ null = nodo raíz                                     │
├─────────────────────┼─────────────────────────────────────────────────────┤
│ category            │ Categoría para ícono:                               │
│                     │  - "security"    → 🔒 validación, seguridad        │
│                     │  - "agent"       → 🧠 intención, razonamiento     │
│                     │  - "connector"   → 🔌 conectores externos          │
│                     │  - "mcp"         → 📦 herramientas MCP              │
│                     │  - "database"    → 🗄️ DB, RAG, graph              │
│                     │  - "llm"         → 🤖 llamadas a modelos           │
│                     │  - "tool"        → ⚙️ herramientas locales         │
│                     │  - "search"      → 🌐 web search                    │
├─────────────────────┼─────────────────────────────────────────────────────┤
│ title               │ Título visible en el árbol                           │
├─────────────────────┼─────────────────────────────────────────────────────┤
│ log                 │ Log técnico interno                                  │
├─────────────────────┼─────────────────────────────────────────────────────┤
│ payload             │ JSON con datos adicionales:                          │
│                     │  - Para started: { "startTime": 1718000000000 }      │
│                     │  - Para otros: datos específicos del paso            │
├─────────────────────┼─────────────────────────────────────────────────────┤
│ duration            │ Duración en ms (solo para completed/failed)         │
├─────────────────────┼─────────────────────────────────────────────────────┤
│ user_friendly_summary │ Texto amigable para mostrar al usuario          │
├─────────────────────┼─────────────────────────────────────────────────────┤
│ error               │ Mensaje de error (si ocurrió)                       │
├─────────────────────┼─────────────────────────────────────────────────────┤
│ timestamp           │ Fecha en ISO 8601 (segundos desde epoch)            │
└─────────────────────┴─────────────────────────────────────────────────────┘
