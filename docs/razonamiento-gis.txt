╔══════════════════════════════════════════════════════════════════════════════╗
║              1. VISIÓN GENERAL DEL FLUJO                                    ║
╚══════════════════════════════════════════════════════════════════════════════╝

  El razonamiento NO es lineal simple (Usuario → LLM → Respuesta).
  Es un pipeline multicapa con routing de herramientas, contexto
  recuperado de SQLite + ChromaDB, y un loop de tool-calling:

  ┌──────────┐   ┌──────────────┐   ┌──────────────┐   ┌────────────┐
  │ Usuario  │──>│  Frontend    │──>│ Tauri Rust   │──>│ Python     │
  │ (Chat)   │   │  React/TS    │   │ send_message │   │ Sidecar    │
  └──────────┘   └──────────────┘   └──────┬───────┘   └─────┬──────┘
                                           │                  │
                                           ▼                  ▼
                                    ┌──────────────┐  ┌──────────────┐
                                    │ SQLite DB    │  │ ChromaDB     │
                                    │ (contexto)   │  │ (embeddings) │
                                    └──────────────┘  └──────────────┘
                                           │
                                           ▼
                                    ┌──────────────┐
                                    │ LLM Router   │──> Ollama / OpenAI
                                    │ (tool-call   │──> Anthropic / OpenRouter
                                    │  loop, max   │──> LM Studio
                                    │  10 iters)   │
                                    └──────┬───────┘
                                           │
                              ┌────────────┼────────────┐
                              ▼            ▼            ▼
                       ┌──────────┐ ┌──────────┐ ┌──────────┐
                       │ Tools    │ │ MCP GIS  │ │ Web      │
                       │ Código   │ │ (qgis)   │ │ Search   │
                       └──────────┘ └──────────┘ └──────────┘
                                           │
                                           ▼
                                    ┌──────────────┐
                                    │ Respuesta    │
                                    │ con trazas   │
                                    │ + fuentes    │
                                    └──────────────┘

║              2. CAPAS DEL SISTEMA                                           ║
╚══════════════════════════════════════════════════════════════════════════════╝

  ─── Capa 1: Frontend (React/TypeScript) ─────────────────────────────────────
  Archivos: src/features/tools/GisToolPanel.tsx
            src/features/workspace/mcp/Mcp*.tsx (8 componentes)
            src/components/chat/ChatPanel.tsx
            src/api/llm.ts, src/api/chat.ts

  El usuario escribe en el ChatPanel. El frontend construye un
  SendMessageInput con: project_id, content, provider, model, endpoint.
  También puede activar web_search y toggles de contexto (RAG, assets, grafo).

  Los GIS Tools se muestran en GisToolPanel.tsx, que lee los 5 tools
  hardcodeados de workspace-data.ts (Buffer espacial, Medir distancia,
  Heatmap, Clustering espacial, Consultar norma POT).

  Los servidores MCP se gestionan desde McpServersPage.tsx, que permite
  registrar servidores (qgis-mcp, memory-mcp, arcgis-mcp, supabase-mcp)
  y ver sus herramientas con schemas JSON.

  ─── Capa 2: Rust Tauri Commands ─────────────────────────────────────────────
  Archivos: crates/geonexus-tauri/src/commands/chat/send_message.rs (391 L)
            crates/geonexus-tauri/src/commands/chat/context.rs (100 L)
            crates/geonexus-tauri/src/commands/chat/tools.rs (306 L)
            crates/geonexus-tauri/src/commands/llm.rs (531 L)
            crates/geonexus-tauri/src/commands/containers_mcp.rs (443 L)

  send_message.rs es el ORQUESTADOR PRINCIPAL. Su flujo interno:

    1. Validar input → crear/conseguir conversación
    2. Guardar mensaje del usuario en SQLite
    3. Recuperar RAG chunks (vía Python sidecar → ChromaDB)
    4. Construir contexto del proyecto (assets + graph_nodes desde SQLite)
    5. Web search opcional (vía Python sidecar → DuckDuckGo/Bing/etc.)
    6. Construir sistema de mensajes con:
       - System prompt + Project context + Web context + RAG chunks
       - Historial de conversación
    7. LOOP DE TOOL-CALLING (máx 10 iteraciones):
       a. Enviar mensajes al LLM vía Python sidecar
       b. Si LLM responde con tool_calls → ejecutar herramienta
       c. Agregar resultado como tool message → continuar loop
       d. Si LLM responde con contenido → break, guardar respuesta
    8. Extraer stats (tokens, duración, tokens/segundo, costo)
    9. Retornar SendMessageResponse

  Las herramientas disponibles en el loop (tools.rs):
    - read_file(ruta) → lee archivo del proyecto
    - search_code(patrón) → busca en código fuente
    - list_directory(ruta) → lista directorio
    - glob_files(patrón) → busca archivos por glob

  containers_mcp.rs gestiona 5 herramientas de gestión de contenedores
  (container_list, container_get, container_search, container_sync,
  container_upload) para conectar fuentes de datos locales y en la nube.

  ─── Capa 3: Python Sidecar ──────────────────────────────────────────────────
  Archivo: ai/sidecar.py (124 L)
           ai/llm/router.py (319 L)
           ai/llm/ollama.py, openai.py, openrouter.py, anthropic.py, lmstudio.py
           ai/memory/chroma.py (158 L)
           ai/context/builder.py (51 L)
           ai/web_search.py (201 L)

  El sidecar atiende 8 acciones vía CLI:
    --action index          → Indexar documento completo
    --action extract        → Solo extraer entities del grafo
    --action ping_llm       → Verificar conectividad con proveedor LLM
    --action chat_llm       → Enviar mensaje al LLM (con tool support)
    --action list_llm_models → Listar modelos disponibles
    --action recall_chunks  → Recuperar chunks relevantes de ChromaDB
    --action build_project_context → Resumir contexto del proyecto desde SQLite
    --action search_web     → Buscar en internet (DuckDuckGo, Google, Bing, SerpAPI)

  El router de LLM (router.py) detecta automáticamente el proveedor según
  el campo provider_type:
    "ollama"    → Ollama (http://localhost:11434)
    "openai"    → OpenAI API
    "openrouter" → OpenRouter
    "anthropic" → Anthropic Claude
    "lmstudio"  → LM Studio (local)

  ─── Capa 4: Base de Datos ────────────────────────────────────────────────────
  SQLite (crates/geonexus-db/):
    - graph_nodes, graph_edges → conocimiento territorial
    - assets, chunks → documentos indexados
    - conversations, messages → historial de chat
    - sync_events → trazabilidad de operaciones

  ChromaDB (ai/memory/chroma.py):
    - Vector store para búsqueda semántica por embeddings
    - Colección "project_memory" por defecto
    - Fallback a JSON file-based mock si ChromaDB no está disponible

  ─── Capa 5: MCP GIS ─────────────────────────────────────────────────────────
  Archivos: src/features/workspace/mcp/mcp-data.ts (190 L)
            crates/geonexus-mcp/src/containers.rs (198 L)

  Servidores definidos en mcp-data.ts:
    qgis-mcp     (online)   → 5 tools GIS (buffer, distance, load_layer,
                              heatmap, cluster) — localhost:7021
    memory-mcp   (online)   → 3 tools de memoria (recall, context, search)
                              — localhost:7011
    arcgis-mcp   (planned)  → Integración futura con ArcGIS
    supabase-mcp (planned)  → Integración futura con Supabase

  El crate geonexus-mcp define el esquema de herramientas para
  contenedores de archivos, pero las herramientas GIS específicas
  (qgis-mcp) se consumen externamente vía HTTP.


╔══════════════════════════════════════════════════════════════════════════════╗
║              3. FLUJO COMPLETO — Escenario: consulta territorial            ║
╚══════════════════════════════════════════════════════════════════════════════╝

  Escenario: El usuario pregunta "¿Qué normas aplican en la Zona Norte?"

  PASO 1 — Usuario escribe en ChatPanel
  Archivo: src/components/chat/ChatPanel.tsx
  Flujo:   onSubmit → sendLlmMessage() o Tauri send_message

  PASO 2 — Frontend construye SendMessageInput
  Datos:   project_id, content="¿Qué normas aplican en la Zona Norte?",
           provider, model, endpoint, use_context=true, web_search=false

  PASO 3 — Rust recibe en send_message (send_message.rs)
  Acciones:
    a. Validar input → crear conversación si no existe
    b. Guardar mensaje del usuario en SQLite
    c. RECALL CHUNKS → Python sidecar --action recall_chunks
       → ChromaDB query con "Zona Norte normas" → top 4 chunks
       → Devuelve textos + scores + asset_ids
    d. PROJECT CONTEXT → Python sidecar --action build_project_context
       → SQLite query: SELECT name, kind FROM graph_nodes WHERE project_id=?
       → SQLite query: SELECT name, kind, status FROM assets WHERE ...
       → Devuelve resumen contextual del proyecto
    e. Construir array de mensajes para el LLM:
       [ system_prompt + project_context + rag_context + historial + user_msg ]

  PASO 4 — Tool-calling loop (send_message.rs)
  Iteración 1:
    a. Enviar al LLM vía Python sidecar --action chat_llm
       → router.py → Ollama (por ejemplo)
       → LLM recibe contexto con nodos del grafo que mencionan "Zona Norte"
       → LLM decide NO llamar tool, responde directamente
    b. Break del loop

  PASO 5 — Rust guarda respuesta + stats
  Acciones:
    a. Guardar mensaje del assistant en SQLite
    b. Extraer stats: tokens, duración, tokens/segundo
    c. Retornar SendMessageResponse al frontend

  PASO 6 — Frontend renderiza respuesta
  Archivo: src/components/chat/ChatPanel.tsx
  Muestra: mensaje del assistant + fuentes (chunks_used, research_sources)


╔══════════════════════════════════════════════════════════════════════════════╗
║              4. FLUJO CON HERRAMIENTAS GIS (futuro)                         ║
╚══════════════════════════════════════════════════════════════════════════════╝

  Actualmente las herramientas GIS están DEFINIDAS pero NO conectadas al
  tool-calling loop del LLM. El flujo FUTURO sería:

  Usuario: "¿Qué zonas están en riesgo de inundación cerca del retiro hídrico?"

  PASO 1-3: Igual que arriba (chat → contexto → RAG)

  PASO 4 — Tool-calling loop con herramientas GIS:
  Iteración 1:
    LLM recibe contexto con nodos "retiro hidrico", "riesgo de inundacion"
    LLM decide llamar: buffer_espacial(coordenadas, radio=500m)
    → Rust ejecuta tool → llama a qgis-mCP (localhost:7021)
    → qgis-mcp devuelve geometría resultante
    → Se agrega como tool message y continúa el loop

  Iteración 2:
    LLM recibe geometría del buffer
    LLM decide llamar: consultar_norma_POT(zona="Norte", tipo="inundacion")
    → Rust ejecuta tool → memory-mcp (localhost:7011)
    → memory-mcp devuelve artículos aplicables
    → Se agrega como tool message y continúa el loop

  Iteración 3:
    LLM tiene contexto completo (buffer GIS + normas)
    LLM responde directamente con análisis integrado
    Break del loop

  PASO 5-6: Igual que arriba (guardar + renderizar)

  Para habilitar esto falta:
    1. Conectar qgis-mcp al tool-calling loop en send_message.rs
    2. Implementar dispatch_tool para herramientas GIS (hoy solo hay
       read_file, search_code, list_directory, glob_files en tools.rs)
    3. El LLM debe recibir las definiciones de herramientas GIS en el
       array de tools del request a la API


╔══════════════════════════════════════════════════════════════════════════════╗
║              5. INTEGRACIÓN CON EL GRAFO DE CONOCIMIENTO                    ║
╚══════════════════════════════════════════════════════════════════════════════╝

  El grafo (graph_nodes + graph_edges) se integra en el razonamiento vía:

  1. ProjectContextPanel.tsx — toggle "Conocimiento territorial"
     → Cuando activo, se incluyen nodos en el contexto del LLM
     → Query: SELECT name AS label, kind FROM graph_nodes LIMIT 8

  2. get_project_context (chat/mod.rs) — Rust query SQLite
     → Retorna ProjectContext { assets: [...], graph_nodes: [...] }
     → Python context/builder.py formatea como texto para el prompt

  3. En el mensaje (Message.nodes_used)
     → Cada respuesta puede registrar qué nodos del grafo se usaron
     → Permite trazabilidad: qué entidades contribuyeron a la respuesta

  4. Visualización en GraphPage.tsx
     → Layout force-directed con d3-force
     → Burbujas arrastrables con persistencia vía update_node_position
     → Búsqueda y filtros por tipo (norma, documento, capa, zona, concepto)
