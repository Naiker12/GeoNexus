╔══════════════════════════════════════════════════════════════════════════════╗
║              1. VISIÓN GENERAL DEL SISTEMA                                  ║
╚══════════════════════════════════════════════════════════════════════════════╝

  El grafo de conocimiento territorial es el sistema que modela, almacena y
  visualiza las relaciones entre los distintos elementos del dominio territorial:

    • Normas (artículos de legislación)
    • Documentos (archivos indexados)
    • Capas GIS (información geoespacial)
    • Zonas (sectores territoriales)
    • Conceptos técnicos (retiro hídrico, zonificación, etc.)

  Propósitos principales:
    1. Visualización interactiva de la red de conocimiento
    2. Contexto trazable para respuestas de IA (citas con evidencia)
    3. Memoria semántica del proyecto para el chat contextual
    4. Exploración de relaciones entre elementos territoriales

  El sistema NO usa Neo4j ni ninguna base de grafos externa. Todo se almacena
  en SQLite con dos tablas relacionales (graph_nodes, graph_edges) con claves
  foráneas y CASCADE deletes.

  FLUJO COMPLETO (indexación de documento):

  1. Usuario sube documento → Rust recibe y guarda el archivo
  2. Rust invoca Python sidecar (index_document_file)
  3. Python extrae chunks de texto + entities del grafo
  4. Python devuelve JSON con graph_nodes y graph_edges
  5. Rust persiste nodos (insert_graph_nodes) y aristas (insert_graph_edges)
  6. El asset se marca con graph_nodes > 0 en assets table
  7. Frontend al abrir Grafo → llama listGraphNodes() + listGraphEdges()
  8. GraphPage renderiza SVG interactivo con burbujas y líneas

  ─── graph_nodes ──────────────────────────────────────────────────────────────

    id           TEXT    PK          -- UUID del nodo
    project_id   TEXT    NOT NULL    -- proyecto al que pertenece
    workspace_id TEXT                -- workspace específico (opcional)
    name         TEXT    NOT NULL    -- nombre visible del nodo
    kind         TEXT    NOT NULL    -- tipo: "norma" | "documento" | "capa" | "zona" | "concepto"
    description  TEXT    NOT NULL    -- descripción del nodo
    evidence     TEXT    NOT NULL    -- referencia a la fuente de extracción
    x            REAL    NOT NULL    -- posición X en el canvas (0-100%)
    y            REAL    NOT NULL    -- posición Y en el canvas (0-100%)
    weight       INTEGER DEFAULT 1  -- peso/tamaño del nodo (escala visual)
    created_at   INTEGER NOT NULL

    Índices: project_id, workspace_id, kind

  ─── graph_edges ──────────────────────────────────────────────────────────────

    id           TEXT    PK          -- UUID de la arista
    project_id   TEXT    NOT NULL    -- proyecto
    source       TEXT    NOT NULL    -- FK → graph_nodes(id) ON DELETE CASCADE
    target       TEXT    NOT NULL    -- FK → graph_nodes(id) ON DELETE CASCADE
    relation     TEXT    NOT NULL    -- tipo de relación: "aplica en", "define", etc.
    strength     INTEGER DEFAULT 50 -- fuerza (0-100) para grosor visual
    created_at   INTEGER NOT NULL

    Índices: project_id, source, target

    NOTA: El CASCADE en source/target asegura que al eliminar un nodo,
          todas sus aristas se eliminan automáticamente.

  ─── assets.graph_nodes ───────────────────────────────────────────────────────

    La tabla assets tiene columna graph_nodes INTEGER DEFAULT 0 que
    cuenta cuántos nodos del grafo están asociados a ese asset.
    Se actualiza en update_asset_indexing_result() tras la indexación.

║              4. PYTHON — EXTRACCIÓN DE ENTIDADES (sidecar)                   ║
╚══════════════════════════════════════════════════════════════════════════════╝

  Archivo: ai/graph/extractor.py (126 líneas)
  Llamado desde: ai/pipeline/indexer.py → step 4

  ─── extract_graph_entities(chunks, project_id, workspace_id) ─────────────────

    Recibe una lista de chunks (fragmentos de texto con page_number) y extrae:

    a) NODOS "norma" — regex: (?:Art\.|Articulo|Artículo)\s*(\d+)
        → Crea nodo "Art. {n}" con weight=2
        → ID: "norma-art-{n}"

    b) NODOS "zona" — regex: (?:Zona|Sector)\s*([A-Z][a-z]+...)
        → Crea nodo "Zona {name}" con weight=3
        → ID: "zona-{name-lowercase}"

    c) NODOS "concepto" — diccionario hardcoded CONCEPT_DESCRIPTIONS
        → Busca palabras clave: "retiro hidrico", "ronda de proteccion",
          "uso industrial", "zonificacion", "drenaje"
        → Cada concepto tiene una descripción fija en español
        → weight=2

    d) ARISTAS entre nodos que co-ocurren en el mismo chunk
        → _cooccurring_pairs() genera todos los pares únicos ordenados
        → _relation_for() determina el tipo según los kinds:
            norma + zona   → "aplica en"
            norma + concepto → "define"
            concepto + zona  → "afecta"
            default         → "asociado con"
        → Todas las aristas se crean con strength=80

    e) FALLBACK: si no se encontró ninguna entidad, crea un nodo "documento"
        único en posición central (50, 50) con weight=2

    NOTA: Las posiciones x/y son aleatorias (random.uniform(20, 80)) en cada
          extracción. No hay algoritmo de force-directed layout — el frontend
          tampoco aplica force simulation. Las posiciones se determinan en
          el momento de extracción y son fijas hasta que se recalcula la red.

  ─── Tipos core (crates/geonexus-core/src/lib.rs:162-186) ────────────────────

    pub struct GraphNode {
        pub id: String,
        pub project_id: String,
        pub workspace_id: Option<String>,
        pub name: String,
        pub kind: String,         // "norma" | "documento" | "capa" | "zona" | "concepto"
        pub description: String,
        pub evidence: String,
        pub x: f64,
        pub y: f64,
        pub weight: i64,
        pub created_at: i64,
    }

    pub struct GraphEdge {
        pub id: String,
        pub project_id: String,
        pub source: String,
        pub target: String,
        pub relation: String,
        pub strength: i64,
        pub created_at: i64,
    }

    Ambos con Serialize/Deserialize (serde) para comunicación con frontend.

  ─── Tauri commands (crates/geonexus-tauri/src/commands/document.rs) ──────────

    list_graph_nodes(project_id) → Vec<GraphNode>
      - SELECT * FROM graph_nodes WHERE project_id = ?
      - Sin filtros adicionales (trae todos los nodos del proyecto)

    list_graph_edges(project_id) → Vec<GraphEdge>
      - SELECT * FROM graph_edges WHERE project_id = ?
      - Trae todas las aristas del proyecto

    rebuild_knowledge_graph(project_id) → ()
      1. clear_graph(project_id) — DELETE FROM graph_nodes WHERE project_id = ?
         (edges se borran por CASCADE)
      2. Scanner todos los assets con status=ready y chunks>0
      3. Para cada asset, crear un nodo "documento" con posición aleatoria
         (random_f64 entre 10 y 90, redondeado a 2 decimales)
      4. Insertar nodos (no crea aristas en rebuild — solo nodos documento)

    NOTA: rebuild_knowledge_graph NO llama al extractor Python. Solo recrea
          nodos tipo "documento" a partir de assets indexados. La extracción
          de entidades (normas, zonas, conceptos) solo ocurre durante la
          indexación de un documento (index_document).

  ─── DB Repository (crates/geonexus-db/src/data_repo.rs) ──────────────────────

    insert_graph_nodes(nodes: &[GraphNode]) → ()
      - INSERT OR REPLACE (upsert por id) en graph_nodes

    insert_graph_edges(edges: &[GraphEdge]) → ()
      - INSERT OR REPLACE (upsert por id) en graph_edges

    list_graph_nodes(project_id) → Vec<GraphNode>
    list_graph_edges(project_id) → Vec<GraphEdge>
    clear_graph(project_id) → ()
      - DELETE FROM graph_nodes (edges se borran por CASCADE)

          verifica list, verifica clear.

║              10. ESTADO ACTUAL Y LIMITACIONES                                ║
╚══════════════════════════════════════════════════════════════════════════════╝

  ✅ IMPLEMENTADO:
    • Schema SQLite con graph_nodes + graph_edges + índices + CASCADE
    • CRUD completo en Rust (insert, list, clear)
    • Extracción de entidades por regex (normas, zonas, conceptos) en Python
    • Aristas por co-ocurrencia en chunks con tipos semánticos
    • Indexación automática durante pipeline de documento
    • 3 comandos Tauri expuestos al frontend
    • Visualización SVG interactiva con burbujas arrastrables
    • NodeSheet con detalle, evidencia, relaciones y peso
    • Partículas animadas en edges (animateMotion SVG)
    • Rebuild de grafo desde assets indexados
    • Integración con chat: ProjectContextPanel + getProjectContext
    • Métricas en DataPanels, DataHeader, AssetSheet
    • Fallback a nodo "documento" si no hay entidades extraídas

  ❌ NO IMPLEMENTADO / LIMITACIONES:
    • Posiciones aleatorias (no force-directed layout) — cada rebuild cambia
      las posiciones, no hay continuity UX
    • rebuild_knowledge_graph solo crea nodos "documento", no ejecuta el
      extractor Python para normas/zonas/conceptos (hay que re-indexar docs)
    • No hay búsqueda de nodos (botón "Buscar nodo" es placeholder visual)
    • No hay filtros (botón "Filtros" es placeholder visual)
    • No hay análisis con IA (botón "Analizar con IA" es placeholder visual)
    • No hay persistencia de posiciones de nodos arrastrados (al recargar
      se pierden porque no se guardan en DB)
    • Python extractor usa solo regex — sin NLP ni embeddings semánticos
    • Solo 5 conceptos hardcoded en CONCEPT_DESCRIPTIONS
    • graph.py es placeholder (sin operaciones de grafo como caminos, bfs, etc)
    • No hay trazabilidad completa de qué nodos se usaron en cada respuesta
      (nodes_ued está definido pero no se implementó el tracking real)
    • No hay deduplicación de nodos entre documentos (si dos docs mencionan
      "Art. 5", se crean dos nodos separados)

  PRÓXIMOS PASOS (roadmap V1.2):
    • Force-directed layout en frontend (d3-force o simulación propia)
    • Persistencia de posiciones arrastradas (guardar x/y en DB)
    • NLP semántico en extractor Python (embeddings + clasificación)
    • Búsqueda y filtros funcionales
    • Análisis con IA de la red (centralidad, clusters, caminos)
    • Deduplicación de nodos entre documentos
    • Graph operations (caminos, subgrafos, bfs) en graph.py
    • Tracking real de nodes_used en respuestas del chat

  Ver datos directo en SQLite:
    sqlite3 data/geonexus.db "
      SELECT g.name, g.kind, g.weight, COUNT(e.id) as edges
      FROM graph_nodes g
      LEFT JOIN graph_edges e ON e.source = g.id OR e.target = g.id
      GROUP BY g.id ORDER BY edges DESC;
    "

  Ver assets con nodos de grafo:
    sqlite3 data/geonexus.db "
      SELECT name, kind, graph_nodes FROM assets WHERE graph_nodes > 0;
    "
