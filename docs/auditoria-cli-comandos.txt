5. FLUJO DE CHAT COMPLETO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Según docs/02-chat-arquitectura.txt, el flujo send_message tiene 15 pasos:

1. Frontend → invoke("send_message", {conversation_id, content})
2. Tauri inserta mensaje usuario en DB
3. Recupera contexto del proyecto (build_project_context)
4. Recupera chunks relevantes (RAG recall_chunks)
5. Recupera historial de mensajes
6. Construye prompt con contexto + RAG + historial
7. Skills activos se inyectan en el system prompt
8. Tools MCP registradas se agregan como function calling
9. Envía a LLM (chat_llm o chat_llm_stream)
10. Procesa respuesta streaming (eventos "delta" al frontend)
11. Si hay tool_calls → ejecuta tools MCP → retroalimenta al LLM
12. Extrae entidades del grafo (extract_chat_entities)
13. Guarda mensaje del asistente en DB
14. Emite evento de conversación completada
15. Retorna respuesta final al frontend

NOTA: Este flujo está diseñado pero NO COMPLETAMENTE implementado.
Partes como tool_calls loop y grafo automático están en desarrollo.

Skills:
- Skills en formato Markdown con frontmatter (YAML)
- Instalación desde archivo .md o GitHub
- Almacenados en app_data_dir/skills/
- Built-in: pot-analyzer (POT colombianos)
- Toggle: activar/desactivar por skill

MCP (Model Context Protocol):
- Protocolo estándar: JSON-RPC 2.0
- Transportes: HTTP, STDIO, SSE (declarado)
- Auto-descubrimiento de tools en servidores HTTP
- Allowlist por tool con rate limiting
- Auditoría completa de llamadas
- Formato LLM: function calling con prefijo mcp__
- Import/export config compatible con Claude Desktop

Coding Agent:
- Workflow: clarify → start_generation → approve_plan → resolve_permission
- Sistema de permisos para ejecutar código
- Estado compartido via PermissionState

Telegram:
- Polling de mensajes desde bot de Telegram
- Pairing via código
- Envío de respuestas y chat actions
- Almacenamiento cifrado de tokens (AES-256-GCM)
