╔════════════════════════════════════════════════════════════════════════════╗
║   ARQUITECTURA GENERAL — FLUJO DE DATOS                                 ║
╚════════════════════════════════════════════════════════════════════════════╝

  Frontend (React/TypeScript) ←→ Tauri IPC (geonexus-tauri commands)
       ↓
  Capa de Comandos (cada archivo en src/commands/)
       ↓
  geonexus-core     → tipos compartidos, eventos, workers, crypto
  geonexus-db       → persistencia SQLite vía sqlx
  geonexus-mcp      → comunicación con servidores MCP externos
  geonexus-fs-mcp   → operaciones de filesystem con seguridad
       ↓
  SQLite (geonexus.db) + ChromaDB (vectores) + Servidores MCP externos

  El flujo típico de un comando IPC:
  1. Frontend llama a invoke("comando", { args })
  2. Tauri deserializa y enruta al handler en geonexus-tauri
  3. Handler usa repositorios (geonexus-db) o servicios (geonexus-mcp)
  4. Resultado se serializa y devuelve al frontend

║   EVENT BUS SYSTEM                                                      ║
╚════════════════════════════════════════════════════════════════════════════╝

  EventBus (geonexus-core/events/bus.rs):
     ├── Tokio broadcast channel (suscripción en tiempo real)
     ├── SQLite persistence (histórico)
     ├── Domain enum + EventType enum (tipado fuerte)
     └── Event forwarder → frontend (events::start_event_forwarder)

  Flujo:
     Worker/Command emite → EventBus → broadcast a suscriptores + SQLite
                                  ↓
                           Event forwarder → Tauri event → Frontend listener

╚════════════════════════════════════════════════════════════════════════════╝

  WorkerPool (geonexus-core/workers/):
     ├── Tokio runtime separado (2 worker threads)

     ├── Dispatch a handler según agent_type

╔════════════════════════════════════════════════════════════════════════════╗
║   MCP ARCHITECTURE                                                      ║
╚════════════════════════════════════════════════════════════════════════════╝

  Dos caras del MCP:
     1. geonexus-mcp crate: Cliente MCP que se conecta a servidores externos
        (registro, ping, execute tool, allowlist)
     2. geonexus-fs-mcp crate: Servidor MCP de filesystem (atiende tools locales)

  Además: containers_mcp module → 5 tools virtuales sobre conectores cloud
  Y: tool_catalog.rs → unifica tools locales + MCP para LLM function calling
