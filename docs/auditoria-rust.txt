FLUJO DE EJECUCIÓN MCP:
1. Frontend llama invoke("call_mcp_tool", {server_id, tool, args})
2. Tauri handler → registry::get_server() → executor::call_tool_for_server()
3. Executor verifica allowlist + rate limit
4. Según transporte: HTTP (handshake + POST) o STDIO (spawn + JSON-RPC)
5. Sanitiza respuesta (elimina secrets)
6. Audita (inserta en mcp_tool_calls)

7. CRYPTO (crypto.rs) — SIN CAMBIOS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Algoritmo: AES-256-GCM
Derivación de key:
1. GEONEXUS_SECRET_KEY → key base64 (32 bytes)
2. Sino → SHA-256("geonexus-telegram-v1:" + machine_id)
   - Windows: HKLM\...\MachineGuid
   - Linux: /etc/machine-id
   - Fallback: "geonexus-fallback-id"
Formato: nonce_base64:ciphertext_base64
Tests: encrypt/decrypt roundtrip, tamper detection, fingerprint

  - CI/CD: windows-latest en GitHub Actions

Linux:
  - AppImage como target de bundle
  - crypto.rs lee /etc/machine-id
  - Sin CI/CD para Linux

macOS:
  - DMG como target de bundle
  - Sin implementación de machine-id (fallback string)
  - Sin CI/CD para macOS

NOTA: El flujo de CI/CD release.yml solo compila para Windows.
