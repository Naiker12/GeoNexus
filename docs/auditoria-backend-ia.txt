1. VISIÓN GENERAL

El backend IA es un sidecar Python (CLI) que Tauri invoca como
subproceso. Su función principal: LLM routing, RAG, knowledge
graphs, embeddings, web search, audio, y extracción GIS.

No es un servidor web — es un dispatcher CLI puro invocado por
la aplicacion Rust mediante `sidecar.py --action <accion> [args]`.

Ruta base: D:\GeoNexus\ai\
Intérprete: .venv\Scripts\python.exe (virtual env)
Entry point: sidecar.py (CLI dispatcher principal)

Características del router:
- Tool calling soportado (function calling) en chat sincrónico y streaming
- Tool calls se agregan incrementalmente durante streaming (por índice)
- Reasoning content (thinking) expuesto via streaming para Anthropic/OpenAI
- Rate limiting para proveedores locales (3K context window, 1K max tokens)
- Fallback a determinista si no hay API key configurada
- Normalización de usage (prompt/completion tokens) entre proveedores

Variables de entorno requeridas según proveedor:
- OPENAI_API_KEY
- ANTHROPIC_API_KEY
- OPENROUTER_API_KEY

5. MEMORIA VECTORIAL (memory/)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

chroma.py:
- Cliente ChromaDB real con fallback a MockChromaClient local (JSON files)
- Mock almacena embeddings en archivos JSON bajo ./chroma_db/
- Operaciones: upsert, query con filtro where
- Cálculo de similitud: coseno (manual en mock, nativo en ChromaDB real)
- Distancia = 1 - similitud

embeddings.py:
- Tres estrategias en orden de prioridad:
  1. OpenAI embeddings (text-embedding-3-*)
  2. Ollama embeddings (http://localhost:11434/api/embeddings)
  3. Determinista (SHA-256 → seed → RNG gauss → normalizar a vector unitario)
- Dimension por defecto: 1536 (OpenAI ada-002)
- Variables de entorno: GEONEXUS_OPENAI_EMBEDDING_MODEL,
  GEONEXUS_OLLAMA_EMBEDDING_MODEL, GEONEXUS_OLLAMA_EMBEDDING_URL

fs_embed.py (CLI independiente):
- Embedding determinista local (SHA-256 como seed)
- Cache en .fs_embed_cache/ (archivos JSON por archivo)
- Comandos: ping, index, search
- Dimensión: 384 (más pequeña que embeddings.py)
- Búsqueda por similitud coseno sobre cache local

6. PIPELINE DE INDEXACIÓN (pipeline/indexer.py)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Flujo completo:
1. extract_text(file_path) → texto plano (reader.py soporta TXT, PDF, DOCX)
2. chunk_text(text) → lista de chunks (docs/chunker.py)
   - chunk_size: 800 palabras, overlap: 150 palabras
   - Preserva marcadores [PAGINA N]
3. generate_embeddings(chunks) → vectores (embeddings.py)
4. upsert_chunks_to_vector_store(ids, embeddings, documents, metadatas) → ChromaDB
5. extract_graph_entities(chunks) → nodos + aristas (graph/extractor.py)

Reader.py soporta:
- .txt → UTF-8 con fallback Latin-1
- .pdf → pypdf con fallback binario (regex sobre bytes crudos)
- .docx → python-docx con fallback

7. KNOWLEDGE GRAPH (graph/)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

extractor.py (desde documentos):
- Detecta: Artículos (Art. N), Zonas (Zona/Sector Nombre), conceptos
  predefinidos (20 conceptos GIS colombianos como "retiro hidrico")
- Aristas: "aplica en", "define", "afecta", "asociado con"
- Posiciones iniciales por tipo (norma, zona, concepto, documento, capa)

chat_extractor.py (desde chat):
- Detecta: patrones de definición ("X es un..."), acrónimos (3-6 mayus),
  términos multi-word con mayúscula, tecnologías comunes (Python, Rust, etc.)
- Stop phrases y stop definitions para evitar falsos positivos
- Aristas entre todos los nodos co-ocurrentes (fully connected por turno)

layout.py:
- Posicionamiento radial por clusters semánticos
- Orden: concepto → norma → zona → capa → documento → chat_turn →
  web_search → upload → connector → rag_recall
- Jitter aleatorio para evitar solapamiento (< 1.5%)

memory.py:
- Algoritmo Ebbinghaus: score = base_weight * e^(-t/stability) + frequency_bonus
- Nodos pinned: score mínimo 5.0
- Decay de peso según días sin uso

13. INTEGRACIÓN CON TAURI (Rust → Python)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

La comunicación es unidireccional:
- Tauri invoca sidecar.py como subproceso vía std::process::Command
- Pasa argumentos CLI: --action <accion> --param1 valor1 --param2 valor2
- Lee JSON de stdout
- La variable GEONEXUS_DB_PATH se pasa al entorno del subproceso
- Tauri espera la respuesta como JSON en una línea de stdout

No hay comunicación bidireccional ni streaming entre Rust y Python
(el streaming se hace desde el LLM directamente al frontend via Tauri events)

Archivos:
- server.py: STDIO MCP server (tools/list, tools/call)
- tools.py: 5 tools definidas (container_list, container_get,

.pdf, .xlsx, .zip, .json, .docx, etc.
