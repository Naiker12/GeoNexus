1.2 Lo que funciona bien
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  - LLM Router: 5 providers con fallback, streaming, tool calls.
    Diseno limpio.
  - MCP Discovery: STDIO y HTTP discovery funcional con JSON-RPC 2.0.
  - Knowledge Graph: Extraccion de entidades basica funcional con
    ChromaDB.
  - Web Search: 4 providers con fallback, deep search multi-query.
  - Tipado TypeScript: Fuerte tipado en frontend, interfaces completas.
  - Temas UI: 8 temas con CSS variables, implementacion solida.
  - Rust modular: 4 crates con separacion de responsabilidades aceptable.

1.3 Lo que esta roto
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  - Telegram: No hay frontend listener. Mensajes entrantes nunca se
    procesan. Pipeline roto.
  - Pipeline duration: Calculo en useChatSession.ts:454 es
    Date.now() - (Date.now() - 0) = 0. Bug matematico.
  - saveConversationId en mount: useChatSession.ts:134 limpia el ID
    guardado al montar. Bug de sesion.
  - MCP handshake versiones inconsistentes: executor.rs usa "2025-06-18",
    pinger.rs y stdio.rs usan "2025-03-26".
  - pinger.rs:74 tiene typo: "2025-03-26}" con llave extra.

в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  - OneDrive OAuth: Placeholder. No implementado.
  - Sistema de eventos backend: Frontend 100%, backend 0%.
  - GIS real: Solo shapefile_extractor.py y geo_utils.py placeholders.
    Sin QGIS, ArcGIS, PostGIS real.
  - Task Queue worker: La queue existe en SQLite, no hay worker loop
    procesandola.
  - Rate limiting en MCP: Campo rate_limit en AllowlistRule nunca se usa.
  - Session/Project/Global Memory: Solo ChromaDB RAG basico.
    Sin jerarquia de memoria.
  - Artifact system: No existe como concepto central.
  - Multi-tenancy: Todo hardcodeado a "project-default".

2.2 Problemas estructurales
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ

  Problema 1: God Command Hub
  в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  geonexus-tauri/src/main.rs registra 90+ comandos en un solo archivo.
  Cada modulo de comandos esta aislado, pero el registro centralizado
  es un punto de acoplamiento masivo.

  Impacto: Dificil de mantener, probar y extender. Cualquier nuevo
  comando toca main.rs.

  Problema 2: Sidecar CLI como cuello de botella
  в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  Cada operacion Python lanza un subproceso, paga el startup de Python
  (~200-500ms), ejecuta, y muere. Para RAG, cada consulta paga
  200-500ms de overhead antes de tocar ChromaDB.

  Impacto: Latencia innecesaria en operaciones frecuentes (RAG, chat).

  Problema 3: Sin capa de abstraccion frontend-backend
  в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  api/chat.ts llama directamente invoke("send_message", {...}).
  No hay bus de eventos formal, cola de mensajes, retry policy,
  ni circuit breaker.

  Impacto: Fallos de red/Tauri se traducen en excepciones no manejadas.

  Problema 4: SQLite como unica base de datos
  в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  Para un sistema multiagente con memoria, eventos y telemetria,
  SQLite no escala. 12 tablas tendran contencion de escritura.

  Impacto: Degradacion con uso intensivo (~100+ consultas/minuto).

  Problema 5: ChromaDB como proceso embebido
  в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  ChromaDB corre dentro del sidecar Python como SQLite local.
  No hay separacion entre memoria del sistema y memoria del usuario.

  Impacto: Mezcla de datos, dificultad para limpiar/resetear memoria.

10.1 Estado actual

    в”ңв”Җв”Җ collection: project_memory
    в”ңв”Җв”Җ query_chunks() вҶ’ RAG por embedding
    в””в”Җв”Җ no jerarquia, no metadata estructurada

  Knowledge Graph (SQLite)
    в”ңв”Җв”Җ graph_nodes (23 campos)
    в”ңв”Җв”Җ graph_edges (source, target, relation)
    в”ңв”Җв”Җ Python extractors: extractor.py, chat_extractor.py
    в””в”Җв”Җ compute_memory_score() en Python

  Frontend:
    в”ңв”Җв”Җ GraphPage + GraphCanvas + NodeSheet
    в”ңв”Җв”Җ GraphFilters + GraphActivityPanel
    в””в”Җв”Җ graph-draw, graph-data, graph-colors

10.2 Problemas
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ

  1. Sin jerarquia de memoria. Todo va a project_memory.
     No hay:
       - Session Memory (lo de esta conversacion)
       - Project Memory (conocimiento acumulado)
       - Global Memory (config, preferencias)
       - Ephemeral Memory (resultados temporales, cache)

  2. Graph y ChromaDB no sincronizados.
     Nodos del graph deberian tener referencias a chunks en
     ChromaDB, pero no hay foreign keys cross-DB.

  3. compute_memory_score() recorre TODOS los nodos.
     SELECT * FROM graph_nodes. Con 10,000 nodos tomaria
     segundos en Python.

  4. Extraccion de entidades es basica (regex). No hay NER
     con modelos. requirements.txt no tiene spaCy.

17. ALGORITMOS RECOMENDADOS

17.1 Intent Classification
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ

  Actual: No existe. Se asume que todo es chat.

  Recomendado:
    1. Generar embedding del mensaje del usuario
    2. Cosine similarity con ejemplos de cada intent:
       - "query"    вҶ’ "que datos hay sobre...?"
       - "analysis" вҶ’ "analiza la correlacion entre..."
       - "planning" вҶ’ "crea un proyecto que..."
       - "research" вҶ’ "investiga sobre..."
       - "coding"   вҶ’ "escribe una funcion que..."
       - "gis"      вҶ’ "muestra en el mapa..."
    3. Threshold > 0.85 вҶ’ intent directo
    4. < 0.85 вҶ’ LLM-based classification
    5. Fallback вҶ’ "query" (chat general)

17.2 RAG con reranking
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ

  Actual: recall_chunks вҶ’ cosine similarity simple, top-K fijo (4).

  Recomendado:
    1. Query expansion: generar 3 variaciones (LLM)
    2. Multi-vector: embedding + BM25 + keyword
    3. Fusion: Reciprocal Rank Fusion (RRF)
    4. Reranking: cross-encoder (BAAI/bge-reranker-v2-m3)
    5. Context window optimization:
       - Presupuesto: 4000 tokens
       - Seleccionar chunks mas relevantes hasta llenar budget
       - Preferir chunks contiguos del mismo documento

17.3 Knowledge Graph incremental
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ

  Actual: SELECT * FROM graph_nodes cada vez.

  Recomendado:
    1. Score decay:
       score = base * e^(-lambda * days_since_access) + 0.1 * visits
    2. Actualizacion incremental:
       - Al leer nodo: UPDATE last_accessed, visit_count++
       - Al crear nodo: INSERT con score = 1.0
       - Cada 100 ops: batch UPDATE de scores decaidos
       - NUNCA recorrer toda la tabla
    3. Graph traversal:
       - BFS limitado a profundidad 3
       - Weighted edges por co-occurrence
       - PageRank personalizado sobre subgraph

17.4 Streaming con backpressure
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ

  Actual: Tauri events sin control de flujo.

  Recomendado:
    1. Token buffer: 100ms window, batch de tokens
    2. Frontend: requestAnimationFrame para render
    3. Backpressure:
       - Si buffer frontend > 500 tokens вҶ’ pausar emision
       - Si buffer frontend < 100 tokens вҶ’ reanudar
    4. Chunked messages para tool calls grandes

17.5 Task Queue con prioridad
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ

  Actual: FIFO simple sin prioridad.

  Recomendado:
    1. Priority queue: 0 (critical) a 3 (background)
       - CRITICAL: user-facing chat responses
       - HIGH: tool calls en pipeline activo
       - MEDIUM: indexacion, sync
       - LOW: cleanup, analysis aggregation
    2. Deadlines: cada tarea tiene max_execution_ms
       - Excedido вҶ’ kill + retry (max 2) вҶ’ failed
    3. Worker pool: N workers tokio (N = num_cpus * 2)
       - Cada worker: pop вҶ’ execute вҶ’ complete/fail
       - Health check: heartbeat cada 5s

================================================================================
RESUMEN EJECUTIVO
================================================================================

GeoNexus es un proyecto ambicioso con una base tecnica solida pero con
desviaciones arquitectonicas significativas.

FORTALEZAS:
  - Stack tecnologico moderno (Tauri 2 + React 18 + Rust + Python)
  - Tipado fuerte en toda la cadena
  - MCP como abstraccion de herramientas (decision correcta)
  - Documentacion interna excelente (22 docs)

DEBILIDADES CRITICAS:
  - Sistema multiagente es sobreingenieria pura (codigo muerto ~30%)
  - GIS no existe (contradice el branding del proyecto)
  - Telegram esta roto
  - Seguridad tiene agujeros (API keys en localStorage, SSRF)
  - Bugs matematicos en el hook central de chat
  - MCP handshake mal implementado (versiones inconsistentes, typo)
  - Sin migraciones de base de datos
  - Frontend monolito en hooks de 600+ lineas
  - Tests insuficientes (81 Rust + 8 Frontend para ~25,000 lineas)

LA DECISION ARQUITECTONICA MAS IMPORTANTE:
  GeoNexus debe ser un Conversational AI Workspace (chat-centric con
  paneles contextuales). No un IDE. No una plataforma GIS tradicional.
  No un CRM de datos. El chat es el centro; GIS, coding, MCP son
  herramientas invocadas desde el chat.

RECOMENDACION FINAL:
  1. Bugs P0 primero (1-2 dias)
  2. Eliminar codigo muerto + refactor hooks (semana 1-2)
