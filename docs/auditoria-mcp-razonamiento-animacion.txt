RESUMEN EJECUTIVO
================================================================================
  GeoNexus tiene tres subsistemas que deberían sentirse como uno solo, pero hoy
  están fragmentados:

    1. MCP Runtime     → Registro, ping, discovery y ejecución HTTP funcional
    2. Razonamiento    → Pipeline real en send_message.rs con eventos Tauri
    3. Animaciones UI  → Componentes visuales parcialmente conectados y duplicados

  El chat principal (send_message) NO usa herramientas MCP. Solo usa 4 tools
  locales de código (read_file, search_code, list_directory, glob_files).
  MCP vive en una vía paralela (agent.rs) que ejecuta todas las tools online
  con argumentos genéricos, sin pasar por el LLM.

  En la UI hay DOS sistemas de timeline de razonamiento (chat vs IDE) y un
  tercer pipeline (useChatPipeline) que escucha eventos que el chat normal no
  emite. Resultado: animaciones genéricas, iconos incorrectos, y componentes
  muertos (ThinkingTextBlock, McpToolCallCard, toolCalls del pipeline).

  Prioridad recomendada:
    P0 — Unificar eventos backend → un solo bus de razonamiento
    P1 — Conectar MCP al tool-calling loop del LLM
    P2 — Rediseñar animaciones (ThinkingCard + ReasoningTimelineBlock)
    P3 — Ejecución STDIO, seguridad tokens, tests

1.1 ARQUITECTURA ACTUAL
--------------------------------------------------------------------------------

  Backend (crate geonexus-mcp):
    crates/geonexus-mcp/src/
      types.rs      — McpServer, McpTool, McpTransport, allowlist
      registry.rs   — CRUD SQLite, auto_discover_tools, audit_tool_call
      pinger.rs     — Ping HTTP/SSE
      executor.rs   — call_tool vía JSON-RPC HTTP (tools/call)
      stdio.rs      — discover_tools (spawn + initialize + tools/list)
      handshake.rs  — Cliente HTTP MCP

  Comandos Tauri (handlers.rs):
    list_mcp_servers, register_mcp_server, delete_mcp_server
    ping_mcp_server, ping_mcp_server_url
    list_mcp_tools, call_mcp_tool
    import_mcp_config, export_mcp_config
    upsert_allowlist, list_allowlist

  Crate adicional geonexus-fs-mcp:
    MCP de filesystem con guards de seguridad (path_guard, rate_guard,
    confirm_gate). Separado del runtime MCP genérico.

  Frontend:
    src/features/workspace/mcp/     — Página servidores, registro, terminal
    src/api/mcp.ts                  — Invocaciones Tauri
    src/types/mcp.ts                — Tipos TypeScript
    src/components/chat/McpConnectCard.tsx — Card inline en respuestas LLM

1.3 BRECHAS Y RIESGOS
--------------------------------------------------------------------------------

  [C-1] CHAT NO USA MCP
  ─────────────────────
  Archivo: crates/geonexus-tauri/src/commands/chat/tools.rs
  get_tool_definitions() expone solo 4 herramientas locales de filesystem.
  execute_tool_call() no delega a geonexus_mcp::executor.

  send_message.rs no importa ni referencia MCP (grep: 0 matches).
  El LLM nunca ve las tools descubiertas en mcp_tools.

  Impacto: El usuario registra servidores MCP, ve tools en la UI, pero el
  chat no puede invocarlas. La promesa "GeoAgents + MCP" no se cumple en el
  flujo principal.

  [C-4] AGENT.RS EJECUTA MCP A CIEGAS
  ────────────────────────────────────
  Archivo: crates/geonexus-tauri/src/commands/agent.rs (líneas 154-211)

  Para cada servidor online, itera TODAS las tools Ready y las llama con:
    args: {"query": goal, "sources": sources}

  Problemas:
    - No hay selección semántica (el LLM no decide qué tool usar)
    - Argumentos hardcodeados ignoran input_schema de cada tool
    - Puede ejecutar decenas de tools innecesarias por request
    - Solo funciona con servidores HTTP (salta stdio por url.is_empty())
    - Coste/latencia impredecible

  [A-5] STDIO NO EJECUTABLE
  ─────────────────────────
  call_mcp_tool (handlers.rs:182-184):
    "No se puede llamar tools en servidores stdio a través de HTTP"

  stdio.rs solo implementa discover_tools, no call_tool_stdio.
  La mayoría de servidores MCP del ecosistema (filesystem, github, etc.)
  son STDIO → registrables pero no invocables.

  [M-5] SEGURIDAD TOKENS
  ──────────────────────
  auth_token se guarda en SQLite plano (migración 20260614000001).
  Stronghold/keychain documentado como pendiente en analisis-mcp-completo.txt.

1.4 FLUJO DE DATOS MCP (estado actual)
--------------------------------------------------------------------------------

  Registro UI → register_mcp_server → SQLite mcp_servers
                    ↓ (si STDIO)
              stdio::discover_tools → mcp_tools

  Ping UI → ping_mcp_server → HTTP handshake → auto_discover_tools

  Chat normal → send_message → tools locales (4) → SIN MCP

  Agent mode → agent.rs → executor::call_tool (HTTP) → todas las tools

  Manual → call_mcp_tool (Tauri) → executor (HTTP only)

SECCIÓN 2 — RAZONAMIENTO (Reasoning Pipeline)
================================================================================

2.1 BACKEND — send_message.rs
--------------------------------------------------------------------------------

  Eventos Tauri emitidos durante send_message:

    reasoning:start     → { session_id }
    reasoning:step      → { id, agent_name, agent_type, status, label,
                          sub_items, duration_ms, started_at, completed_at }
    reasoning:sub_item  → { step_id, text }
    reasoning:end       → { session_id, total_ms }

  Pasos emitidos (orden típico):
    1. intent        — classify_intent() → QueryIntent (6 tipos GIS)
    2. graph_context — nodos/aristas del grafo (si hay)
    3. rag           — recall_chunks ChromaDB (top_k=4)
    4. web_search    — si web search activo
    5. skills        — si hay skills seleccionadas
    6. generating    — tool-calling loop LLM (running → success)
       └── sub_items: "Called tool: X (Nms)", "N tokens generados"

  Otros eventos paralelos (no unificados):
    chat:stream_event  → knowledge_lookup, deep_research, generating, tool_call
    llm:tool_call      → { tool_name, tool_call_id, args }
    llm:tool_result    → { tool_name, success, duration_ms }
    llm:done            → { content_length, model }
    chat:preview_chunk  → chunks RAG para preview

  Archivo reasoning core: crates/geonexus-core/src/reasoning.rs
    Solo define QueryIntent (6 variantes GIS) + AnalysisSession + ValidationResult.
    No hay lógica de orquestación aquí; vive en send_message.rs.

2.2 BACKEND — agent.rs (vía alternativa)
--------------------------------------------------------------------------------

  Emite agent:event (NO reasoning:*) con fases:
    discovery → knowledge → mcp → reasoning → result

  useChatPipeline escucha agent:event con STATUS_TO_KIND mapping.
  Este flujo es del modo agente multi-fase, NO del chat estándar.

  [C-2][C-3] DESCONEXIÓN
  El chat normal emite reasoning:* pero useChatPipeline escucha agent:event.
  Resultado: pipeline siempre null en chat → AgentLifeIndicator solo muestra
  "thinking" genérico sin metadata de pasos.
